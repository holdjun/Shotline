# 色彩科学基础

数字影像处理中的色彩科学核心概念：编码、色彩空间、tone mapping、色彩空间转换、LUT。

## 1. 色彩空间 × 编码（两个正交维度）

图像的"色彩"由两个独立维度决定：

- **Encoding（传递函数 / Transfer Function）** — 亮度值怎么映射成数字
- **ColorSpace（色域 / Primaries）** — R/G/B 通道各代表什么颜色

两者正交，自由组合。混淆二者是色彩管理中最常见的错误。

### Encoding（传递函数）

传递函数定义"物理光强"和"存储数值"之间的映射关系。不同传递函数为不同目的优化。

| Encoding | 数学关系（简化） | 设计目的 | 典型场景 |
|----------|-----------------|----------|----------|
| **Linear** | `v = L` | 值 ∝ 光子数，物理精确 | 渲染引擎、合成、RAW 处理中间态 |
| **sRGB** (IEC 61966-2-1) | `v ≈ L^(1/2.4)` | 感知均匀 + 兼容 CRT gamma | Web、消费显示器、JPG/PNG |
| **Gamma 2.2** | `v = L^(1/2.2)` | sRGB 的简化近似 | 老旧工作流、部分游戏引擎 |
| **Gamma 1.8** | `v = L^(1/1.8)` | ProPhoto RGB 标准传递函数 | ProPhoto TIFF 归档 |
| **Rec. 709** | `v ≈ L^0.45`（有效 gamma ≈ 2.22） | 广播电视标准，与 sRGB 近似但不完全相同 | HDTV、SDR 视频 |
| **S-Log2 / S-Log3** | 对数曲线 | 最大化传感器动态范围保留 | Sony 电影摄影机 |
| **Log-C** | 对数曲线 | 同上 | ARRI Alexa |
| **V-Log** | 对数曲线 | 同上 | Panasonic |
| **PQ** (ST 2084) | 感知量化，峰值 10,000 nits | 绝对亮度映射，HDR 显示 | HDR10、Dolby Vision |
| **HLG** (Hybrid Log-Gamma) | 低段 gamma + 高段对数 | HDR/SDR 向后兼容 | HDR 广播 |

几个关键区别：

- **sRGB vs Gamma 2.2**：sRGB 在暗部（≤ 0.0031308）有一段线性区间 `v = 12.92 × L`，避免极暗处的量化噪声；纯 gamma 2.2 没有。二者在中高亮度几乎相同，但不能互换——细微的暗部差异在 8-bit 下会产生可见色阶
- **Log 族**：将传感器的 14+ stops 动态范围压缩到 [0, 1]，暗部和高光细节都保留在数值中，代价是看起来"灰片"（低对比低饱和），必须后期做 tone mapping 还原
- **PQ vs HLG**：PQ 是绝对亮度编码（值 0.5 = 某个固定 nits），HLG 是相对亮度（类似传统 gamma 的思路），HLG 在 SDR 设备上可以直接显示（降级但不破碎），PQ 不行

同一束光在不同编码下的数值：

```
物理亮度 18%（中灰）：
  Linear   → 0.180
  sRGB     → 0.461    (= 1.055 × 0.18^(1/2.4) − 0.055)
  Gamma2.2 → 0.456    (= 0.18^(1/2.2))
  S-Log3   → 0.410    (IRE 域)
  PQ       → 0.385    (假设环境亮度映射到 ~100 nits)

物理亮度 50%：
  Linear   → 0.500
  sRGB     → 0.735
  Gamma2.2 → 0.730
```

### ColorSpace（色域）

色域定义 R/G/B 三原色在 CIE xy 色度图上的坐标，决定能表示的颜色范围。三角形越大，色域越宽。

| 色域 | 覆盖可见光 | 白点 | 典型用途 |
|------|-----------|------|----------|
| **sRGB** (Rec. 709) | ~35% | D65 | Web、消费显示器、JPG |
| **Display P3** | ~45% | D65 | Apple 设备、iPhone 照片 |
| **Adobe RGB** | ~50% | D65 | 印刷（CMYK 覆盖更好） |
| **Rec. 2020** | ~75% | D65 | UHD/HDR 视频 |
| **ProPhoto RGB** | ~90% | D50 | RAW 处理、归档 |
| **ACES AP0** | ~100% | D60 | 电影工业交换标准 |

### 常见组合及含义

| 编码 | 色域 | 含义 | 场景 |
|------|------|------|------|
| Linear + ProPhoto | 线性光 × 最宽色域 | 最大信息保留，物理精确计算 | RAW 处理 |
| Linear + sRGB | 线性光 × 标准色域 | 渲染中间态 | 3D 渲染、合成 |
| sRGB + sRGB | 感知编码 × 标准色域 | 最终交付，人眼友好 | Web、JPG |
| Gamma 1.8 + ProPhoto | ProPhoto 标准传递函数 | 宽色域非线性存储 | 16-bit TIFF 归档 |
| PQ + Rec. 2020 | HDR 绝对亮度 × 宽色域 | HDR 视频标准 | HDR10 |
| HLG + Rec. 2020 | HDR 相对亮度 × 宽色域 | HDR 广播标准 | NHK/BBC HDR |
| S-Log3 + S-Gamut3.Cine | Sony Log + Sony 宽色域 | 机内录制格式 | Sony 电影摄影 |
| sRGB TRC + ProPhoto | **无交换标准** | 内部管线可用，不应用于文件交换 | 见下方说明 |

### sRGB 编码 + ProPhoto 色域：没有标准，但并非不可用

sRGB 传递函数（IEC 61966-2-1）是为 sRGB 色域 + D65 白点定义的完整标准，没有任何规范定义"sRGB TRC + ProPhoto primaries"这个组合。但这不意味着它在内部处理管线中不能出现：

- Lightroom 的 Develop 模块工作在 ProPhoto 色域 + 类似 sRGB 的感知编码下
- darktable 的 display-referred 模块可以在 Rec. 2020 色域 + sRGB TRC 下工作
- 在内部管线中，传递函数只是一个数学函数（分段 gamma），可以应用于任何数据

关键限制是**文件交换** — 如果你把这个组合写入 TIFF/PNG 并嵌入 ICC Profile，没有标准 profile 能描述它，其他软件无法正确解码。ProPhoto 有自己的标准传递函数（Gamma 1.8），文件交换应使用它。

同理，Log 编码通常和对应的色域配对使用（S-Log3 + S-Gamut3、Log-C + ARRI Wide Gamut），但这是约定而非物理限制。

## 2. 为什么图像处理必须在 Linear 编码下进行

### 线性光的物理含义

Linear 编码中，数值直接正比于光子数量：

- 值 0.5 = 场景中一半的光
- 值 × 2 = 光强翻倍 = 曝光 +1 EV

所有基于物理量的运算（曝光补偿、白平衡、暗角校正、alpha 混合）在 Linear 下只需简单乘法/加法，结果物理正确。

### 非线性编码做数学运算的错误

在 sRGB gamma（或任何非线性编码）下做相同运算会得到物理错误的结果。

**混合偏暗**：将黑（0.0）和白（1.0）各 50% 混合：

```
Linear 空间：
  (0.0 + 1.0) / 2 = 0.50 → 物理亮度 50% ✓

sRGB gamma 空间（简化为纯 gamma 2.4）：
  (0.0 + 1.0) / 2 = 0.50 → 解码回线性 ≈ 0.217
  实际亮度只有 ~22%，视觉上明显偏暗 ✗
```

这就是为什么在 gamma 空间做图层混合、渐变、模糊时会出现"暗带"或"暗缝"伪影。

**曝光非线性**：将亮度乘以 2（+1 EV）：

```
Linear 空间：
  0.18 × 2 = 0.36 → 亮度翻倍 ✓

sRGB gamma 空间（简化为纯 gamma 2.4）：
  0.461 × 2 = 0.922 → 解码回线性 ≈ 0.838
  实际亮度 = 0.838 / 0.18 = 4.66 倍，远超 2 倍 ✗
```

**模糊/缩放偏暗**：高斯模糊本质是像素加权平均。如果在 gamma 空间做，亮像素贡献被低估（gamma 压缩了亮部），结果系统性偏暗。正确做法是先转 Linear → 模糊 → 转回 gamma。

### 哪些操作必须在 Linear 下、哪些不必

| 操作 | 需要 Linear？ | 原因 |
|------|--------------|------|
| 曝光补偿 | 是 | 乘法必须在线性光下才是物理正确的 EV 调整 |
| 白平衡（通道增益） | 是 | 同上 |
| 暗角校正 | 是 | 像素亮度乘法 |
| alpha 混合 / 图层合成 | 是 | 加法混合需要线性才不会出暗缝 |
| 高斯模糊 / resize | 是 | 加权平均在 gamma 空间会偏暗 |
| Tone mapping | 是（输入） | S 曲线设计基于线性光输入 |
| 锐化（USM） | 通常否 | 在感知空间做更符合视觉感知 |
| 色彩分级（LUT） | 取决于 LUT | LUT 烘焙的输入空间决定 |
| AI 模型（降噪、超分） | 取决于训练数据 | 大多数模型训练数据是 sRGB |

## 3. Tone Mapping（色调映射）

### 动态范围的差距

| 来源 | 动态范围 | 说明 |
|------|----------|------|
| 真实场景（日光下） | ~20 stops（1:1,000,000） | |
| 相机传感器（14-bit RAW） | ~14 stops | 14-bit 是量化精度，实际 DR 取决于传感器噪底 |
| SDR 显示器 | ~10 stops（对比度 ~1000:1） | 好的 IPS 面板可达 ~11 stops |
| 8-bit JPG | ~8 stops 有效精度 | 8-bit 量化在暗部可见色阶，限制可用 DR |
| HDR 显示器（PQ 10,000 nits） | ~17 stops | |

RAW 数据的亮度范围远超 SDR 显示设备能力。Tone mapping 的职责是把高动态范围压缩到目标显示范围，同时保持视觉自然。

### 相机 JPG 已经做了 tone mapping

相机直出 JPG/HEIF 并非"无处理"，内部流水线包含：

1. RAW decode + demosaic
2. 白平衡
3. **色调曲线**（Canon Picture Style / Fuji Film Simulation / Sony Creative Look / Nikon Picture Control）
4. 色域转换 → sRGB
5. JPEG 压缩

第 3 步就是 tone mapping。不同相机厂商的色调曲线是其"色彩科学"的核心差异化卖点 — Fuji 的"胶片感"、Sony 的肤色、Canon 的"佳能色"本质上都是不同的 tone mapping + 色彩偏移策略。

### 常见 tone mapping 算法

| 算法 | 类型 | 特点 |
|------|------|------|
| **Reinhard** | 全局 | `L / (1 + L)`，简单温和，高光压缩不够 |
| **Hable filmic** (Uncharted 2) | 全局 S 曲线 | toe 提暗部 + shoulder 压高光，电影感 |
| **ACES filmic** | 全局 S 曲线 | 工业标准，RRT + ODT 分离 |
| **Reinhard-Jodie** | 全局 + 亮度保护 | 在 Reinhard 基础上混合亮度通道，减少色相偏移 |
| **局部 tone mapping** | 局部 | 分区域调整（Lightroom 的高光/阴影滑块本质是局部 TM） |

#### Hable filmic S 曲线

经典的全局 tone mapping 算法（John Hable, GDC 2010），来自 Uncharted 2：

```
f(x) = ((x·(A·x + C·B) + D·E) / (x·(A·x + B) + D·F)) − E/F
```

S 曲线的两个关键区域：

- **Toe（暗部）**：轻微提亮暗部，增加暗部细节可见性
- **Shoulder（高光）**：渐进压缩高光，避免硬裁切白死

White point W 定义"完全白"对应的线性输入值，所有 ≥W 的值映射到 1.0。

### 必须在 Linear 输入上做

Tone mapping 算法期望输入是线性光值（值 ∝ 光强）。如果输入已经是 gamma 或 Log 编码：

- S 曲线的 toe/shoulder 行为会被已有的非线性扭曲
- 高光压缩区域错位，无法正确还原 HDR 细节
- 等效于对数据做了两次非线性变换，结果不可预测

这是编码（Encoding）要求 — 必须是线性输入。对色域没有硬性要求（ProPhoto Linear 和 sRGB Linear 都可以做 tone mapping），但色域会影响 per-channel tone mapping 的去饱和行为（见下节）。

### per-channel tone mapping 的色相偏移

大多数 tone mapping 算法对 R/G/B 三通道独立应用相同函数。当某个通道值远高于其他通道时，该通道被 shoulder 压缩更多，导致通道比例改变 → 色相偏移。

典型表现：高饱和暖色（红/黄日落）在 tone map 后偏向白色（desaturation）。

缓解方法：

- **亮度映射法**：先计算亮度 L，对 L 做 tone map 得到 L'，然后按 L'/L 缩放 RGB — 保持色相不变，但可能产生超范围值
- **宽色域处理**：在 ProPhoto 等宽色域下，同样的饱和色通道值差异更小（色域更大，不需要极端的通道值来表达），shoulder 压缩的不均匀程度降低
- **ACES 方法**：在 tone map 前先做 gamut mapping 到安全范围，避免极端通道值

## 4. 色彩空间转换

### 转换原理

两个 RGB 色彩空间之间的转换路径：

```
源 RGB → (源 RGB→XYZ 矩阵) → CIE XYZ → (色适应，如需要) → CIE XYZ' → (XYZ→目标 RGB 矩阵) → 目标 RGB
```

实际操作中三步合并为一个 3×3 矩阵乘法：`目标_RGB = M × 源_RGB`。

### 宽色域 → 窄色域（例：ProPhoto → sRGB）

转换涉及两步（内含在合并矩阵中）：

1. **色适应（Chromatic Adaptation）**：不同白点意味着"纯白"在物理上对应不同的光谱分布，人眼会自动适应环境光（在钨丝灯下白纸看起来仍然是白的），色适应矩阵模拟这个人眼行为。ProPhoto 使用 D50，sRGB 使用 D65，Bradford 变换是最常用的色适应模型
2. **原色变换**：将源 RGB 原色坐标映射到目标 RGB 原色坐标

如果两个色彩空间使用相同白点（如 sRGB 和 Display P3 都是 D65），则不需要色适应，只有原色变换。

### 窄色域 → 宽色域（例：sRGB → ProPhoto）

反向转换 `ProPhoto = M⁻¹ × sRGB` 数学上无损，但不增加信息 — sRGB 数据只覆盖了 ProPhoto 色域的一小部分，空出来的范围没有真实数据填充。类似于将 MP3 转成 WAV 不会恢复丢失的高频信息。

### 色域外颜色处理（Gamut Mapping）

宽色域转换到窄色域时，存在目标色域无法表示的颜色（如 ProPhoto 中的激光绿在 sRGB 中无法表达）。转换后这些颜色的 RGB 值会落到 [0, 1] 之外，必须处理：

| 方法 | 做法 | 特点 |
|------|------|------|
| **Clip（相对色度映射）** | 直接裁切到 [0, 1] | 简单快速，色域内颜色精确保留，色域外颜色丢失饱和度并可能色相偏移 |
| **感知映射（Perceptual）** | 等比压缩整个色域使其适配目标 | 保持颜色之间的关系，但色域内颜色也被轻微影响 |
| **饱和度映射（Saturation）** | 优先保留饱和度 | 商业图表用，摄影不用 |

大多数 RAW 处理器使用 clip。对于已经经过 tone mapping 的数据，极端色域外值很少出现，clip 通常足够。

ICC Profile 驱动的打印工作流通常使用感知映射，因为打印色域形状不规则（不是简单的三角形），感知映射能更好地利用纸张 + 墨水的实际色域。

### 转换矩阵的生成

手动计算两个色域之间的转换矩阵需要：

1. 查找两个色域的原色 CIE xy 坐标和白点
2. 计算各自的 RGB→XYZ 矩阵
3. 如果白点不同，构建色适应矩阵（Bradford / Von Kries / CAT02）
4. 合并为一个 3×3 矩阵

使用成熟的色彩科学库（如 Python `colour-science`、C++ `OpenColorIO`）可以一行代码生成精确矩阵，避免手动计算中的精度和标准引用错误。

## 5. LUT 与滤镜

### LUT 是什么

LUT（Look-Up Table）是颜色替换查找表 — 输入一个 RGB 值，查表得到一个新的 RGB 值。

| 类型 | 结构 | 功能 | 大小 |
|------|------|------|------|
| **1D LUT** | 3 条独立曲线（R/G/B 各一条） | 对比度、亮度、单通道色偏 | 很小（几 KB） |
| **3D LUT** | 3D 网格（通常 33³ 或 65³ 个格点） | 任意颜色映射，包括交叉通道影响 | 较大（几百 KB ~ 几 MB） |

3D LUT 的工作方式：将 RGB 色彩空间划分为立方体网格，每个格点存储一个目标 RGB 值，输入值落在格点之间时做三线性插值。本质上是一个离散化的"任意颜色函数"。

1D LUT 无法做交叉通道映射（R 通道的输出只取决于 R 通道的输入），所以无法实现色调分离（高光偏暖 + 阴影偏冷）等效果 — 这些需要 3D LUT。

### LUT 被烘焙在固定输入空间

LUT 的制作者在特定色彩空间下设计颜色映射。**LUT 不携带输入空间信息**，使用者必须自己确保输入匹配。

市面上的 `.cube` LUT 主要有两类：

- **创意 LUT**（99%）：假设输入是 **sRGB gamma + sRGB 色域**，用于照片/视频调色
- **技术 LUT**：Log → Rec. 709 转换（如 Sony S-Log3 → Rec.709），用于视频后期

如果把 Linear 编码的数据送入一个为 sRGB gamma 设计的 LUT：

```
LUT 以为 0.5 = 中灰（sRGB gamma 下确实约为中灰）
实际 Linear 0.5 = 物理 50% 亮度 → sRGB gamma 下应该是 0.735
→ LUT 对 0.5 的映射被应用到了错误的亮度位置
→ 整个颜色映射错位：暗部被过度处理，高光被欠处理
```

### 同一个 LUT 对不同来源图像效果一致的前提

要让 LUT 在不同来源（RAW 处理、JPG、视频帧）上产生一致的效果：

1. **输入编码一致** — 数据必须在 LUT 期望的传递函数下（通常 sRGB gamma）
2. **输入色域一致** — 数据必须在 LUT 期望的色域下（通常 sRGB primaries）
3. **动态范围已归一化** — 值域在 [0, 1] 内，无超范围值（RAW 数据必须先经过 tone mapping）

只要这三个条件满足，无论原始数据来自 14-bit RAW 还是 8-bit JPG，LUT 看到的输入空间相同，输出结果一致。
